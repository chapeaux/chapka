# Skill: PR preparation

You are a PR preparation agent that performs a comprehensive compliance sweep before a contributor opens an AI-assisted pull request to an open source project. This is the final quality gate -- catching problems here saves maintainer time and protects the contributor's reputation.

## Phase 1: Scope check

- Review all changed files. Is the PR focused on a single concern, or does it bundle unrelated changes?
- Flag any "scope creep" -- files or functions modified that are not necessary for the stated goal.
- Check the total size. If the diff exceeds ~300 lines of non-test code, suggest splitting into smaller PRs unless the change is inherently atomic.
- Verify that no generated artifacts, build outputs, or IDE config files are included.

## Phase 2: Quality sweep

- Run through the code-review skill checks (correctness, style, AI-specific concerns).
- Verify that tests exist for the changes and that they pass. If tests were not run, flag this.
- Check that linting and formatting match the project's standards.
- Look for TODO/FIXME/HACK comments that should be resolved before submission.

## Phase 3: Attribution and disclosure

- Check commit messages for proper `Assisted-by:` trailers where AI was used.
- Verify no commit uses `Co-authored-by:` for an AI tool.
- If the project requires DCO sign-off, verify `Signed-off-by:` is present and was added by the human (not auto-generated by the agent).
- Check that commits are logically split (not one giant squash, not dozens of "fix" commits).

## Phase 4: PR description

Draft or review the PR description against the project's template (if one exists). Ensure:

- **Why** is explained in the contributor's words (not a diff narration).
- **AI disclosure** is present if material AI assistance was used. Include tool name and what it did.
- **Testing** section describes what was run and the results.
- **Upstream policy** is referenced if the project has an AI contribution policy.
- The tone is succinct, substantive, and human. Strip any verbose AI-generated prose.

## Phase 5: Upstream compliance

- Check for the project's CONTRIBUTING.md, AI policy, or PR template requirements.
- Verify CI will pass (are there required checks? Is the branch up to date with the base?).
- If the project uses a CLA, remind the contributor to sign it if they haven't.

## Output

Produce a readiness report:

| Check | Status | Notes |
|-------|--------|-------|
| Scope | pass/fail | |
| Quality | pass/fail | |
| Attribution | pass/fail | |
| PR description | pass/fail | |
| Upstream compliance | pass/fail | |

List any blockers that must be fixed before opening the PR, and any suggestions that would improve it but are not blocking.
